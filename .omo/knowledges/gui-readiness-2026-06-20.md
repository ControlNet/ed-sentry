# GUI readiness - 2026-06-20

- Prior plan memory says the long-term presentation modes are CLI, Matrix bot, Web, and Tauri desktop. Completed work covers CLI plus Matrix bot only; Web/Tauri GUI work has not started.
- Current crate exposes reusable library modules through `src/lib.rs`, so GUI work should reuse `ed_sentry::{config,event,journal,mission,monitor,state,notifier,delivery}` instead of shelling out to the CLI.
- `EventMonitor` is the current producer boundary. It mutates `SessionState` and returns `Vec<Notification>` from `process_event`, `check_warnings_at`, `start_monitor`, and `finish`.
- `SessionState` is the public AFK session model for dashboard consumers: commander, ship, system, mode, active session, session timestamps, shields/hull/fighter state, scan/kill counts, bounty/merits totals, mission totals, faction kill totals, and rate helpers.
- `MissionTracker` is intentionally separate from `SessionState` and is the right source for dashboard-style mission lists/progress. Do not add GUI aggregation directly to parser types.
- Parser and event model are GUI-friendly because typed promoted events preserve full raw Journal JSON through `raw: Option<serde_json::Value>` and `JournalEvent::raw_payload()`.
- Journal ingestion options already exist for GUI prototypes: `preload_journal_file`, `stream_journal_file`, `LiveTail::from_preload`, and `LiveTail::poll`.
- Replay must remain terminal-only and Matrix-free. A GUI replay screen can reuse the parser/state/monitor pipeline, but should not call the CLI replay path blindly if it would inherit terminal-only assumptions.
- Matrix delivery remains optional, best-effort, watch-mode only, and unencrypted-room only. GUI settings must keep token handling private and must not log or commit `config.toml`.
- No frontend/Tauri structure exists yet. Before implementing components, create a project design system (`DESIGN.md`) and choose the GUI shell strategy explicitly.
- WebUI and Tauri frontend pages should be shared as one web application wherever possible. Keep screens/components/state presentation portable, and isolate environment-specific behavior behind adapters such as Journal file access, config persistence, native dialogs, window controls, and process/background-watch lifecycle.
- Preferred split for future GUI work: reusable frontend app + domain/view models + shell adapters. Tauri should call Rust commands directly for local Journal/config access; browser WebUI should use HTTP/WebSocket or another remote bridge if added later.
- Full-stack WebUI is a valid target. In that mode, the browser frontend remains shared, while a local or hosted backend owns Journal file reading, watch tasks, config access, and event streaming. Keep that backend as a thin transport layer over the same Rust application services used by Tauri, not a second implementation of monitor logic.
- For full-stack WebUI, prefer HTTP for commands/config snapshots and WebSocket or SSE for live monitor events/status updates. Treat local Journal paths and Matrix tokens as private backend-side data; never expose raw token values to the frontend.
- Packaging direction under consideration: keep the existing `ed-sentry` CLI executable as the owner of CLI plus optional embedded WebUI server, and build the desktop GUI as a separate executable. This can let the CLI serve a local full-stack WebUI without forcing users to install the Tauri app, while the Tauri GUI can reuse the same frontend and Rust application services.
- If the current CLI binary embeds WebUI, preserve existing default watch/replay behavior. Prefer an explicit opt-in flag or mode for the server path, and decide carefully because the current public CLI contract has no subcommands.
- Updated WebUI startup preference: do not require a dedicated `--webui` flag for normal use. Add a `[web]` config section; when `[web] enabled = true`, watch mode starts the embedded WebUI server alongside the existing terminal/Matrix behavior.
- User confirmed the `[web] enabled = true` config-driven WebUI startup model as the preferred direction.
- `[web] enabled = true` should not make replay initialize WebUI unless explicitly redesigned later. Keep replay deterministic and terminal-only by default, matching the existing Matrix replay isolation rule.
- WebUI config should default to safe local-only serving, for example `host = "127.0.0.1"`, an explicit/default port, and no public network bind unless the user deliberately changes config. Do not expose Matrix access tokens or raw private config values to the frontend.
- User accepted the recommended GUI defaults: first milestone is live dashboard only, frontend is React + Vite + TypeScript, Web backend is Axum, realtime transport is WebSocket, styling foundation includes Tailwind, design direction is a dark operational dashboard, app architecture uses a shared Rust application service, `ed-sentry.exe` owns optional config-enabled WebUI in watch mode, and the separate desktop binary name should be `ed-sentry-gui`.
- UI component strategy decision: use shadcn/ui for the first GUI implementation. Treat shadcn/ui as copied component source scaffolding over Radix/Tailwind, not as a visual identity. Retheme it through `DESIGN.md` tokens and custom `ed-sentry` dashboard components so the app does not look like a generic shadcn admin template.
- Event history decision: maintain the recent event buffer in the backend, not only in frontend memory, so a newly opened WebUI/Tauri view can immediately display earlier notifications/events from the current process. A fixed-size ring buffer is suitable for the first milestone.
- Config editing decision: the GUI should support editing relevant configuration, not just viewing sanitized status. Editing must still protect secrets: never echo Matrix access tokens back to the frontend as raw values, and write local config through backend/Tauri-safe commands.
- Replay UI decision: do not include replay in the GUI scope. Existing CLI replay stays available and terminal-only.
- Runtime entry decision: CLI and desktop GUI should be different entry points over the same runtime/application service behavior. If config enables `[web]` or `[matrix]`, both the CLI entry and `ed-sentry-gui` entry should be able to start the corresponding WebUI server and Matrix delivery where the active mode supports them. Avoid making Tauri a separate feature universe.
- Tauri service implication: `ed-sentry-gui` should not be treated as only a frontend wrapper. It should bootstrap the same Rust app services as the CLI, honor config-enabled Web/Matrix behavior, and expose the shared frontend through Tauri-native adapters while the optional Web server remains available when `[web] enabled = true`.
- User accepted the remaining recommended implementation defaults: use `pnpm`, place shared frontend under `ui/`, compile the Web server into `ed-sentry` by default while gating runtime startup by `[web]`, serve `ui/dist` in the first phase and defer static asset embedding/release packaging, treat WebUI startup failure as a warning while CLI/GUI monitoring continues, keep first milestone local-first with no auth and warnings for non-localhost bind addresses, use Zustand for frontend dashboard state, avoid a chart library in the first milestone, create root `DESIGN.md`, send raw values plus display text for CLI-parity-sensitive fields in DTOs, and defer historical database/start-stop workflow complexity beyond the live dashboard milestone.
- `ui-ux-pro-max` design lookup for `desktop monitoring dashboard space simulation tactical professional` recommends a real-time monitoring dashboard with OLED dark mode, high contrast, green positive indicators, and Fira Code/Fira Sans typography.
- Adapt that recommendation to this project as an operational dashboard, not a landing page: first screen should show live session health, current commander/ship/system, kill/scan rates, warnings, mission progress, event feed, Matrix status, and Journal source controls.
