# Journal/WebUI Data Gap Audit

Date: 2026-06-21
Scope: current webui data contract versus a local personal Elite Dangerous Journal folder.

## Summary

The current webui can reliably show the existing AFK-monitor domain model: commander, ship, system, mode, session activity, kills, cargo scans detected by the current ReceiveText heuristic, bounties, shield/hull/fighter status, tracked massacre/trade missions, recent notifications, journal source status, and Matrix/Web service status.

The main limitation is not that the local Journal is sparse. The local Journal is rich: 80,428 parsed lines, 145 event types, and 0 malformed lines. The limitation is that `AppSnapshot` is intentionally narrow. It exposes only normalized monitor state, mission state, notification/event-feed history, source metadata, and service status. It does not expose most raw Journal data or any sidecar JSON data.

## Currently Obtainable In WebUI

These are obtainable with the current backend and current Journal sample:

- Session identity/status: commander, ship, system, game mode, active/ended/idle status, elapsed time.
- Combat counters: kills, bounty total, last kill time, total/recent kill rate. In the sample this is backed by 3,040 `Bounty` events. There are no `FactionKillBond` events in the sample.
- Cargo-scan counters: scan count, last scan time, total/recent scan rate. This is not every `ReceiveText`; it only increments for NPC messages matching the current cargo-scan signature. The sample has 3,893 matching messages.
- Ship health: shield up/down from 12 `ShieldState` events and hull percent from 36 `HullDamage` events.
- Fighter state: fighter launched/destroyed/hull from 18 `LaunchFighter`, 13 `FighterDestroyed`, and fighter `HullDamage` records.
- Mission cards: active/completed totals plus mission rows for accepted missions, with massacre kill progress and trade delivery progress. This is backed by `MissionAccepted`, `MissionRedirected`, `MissionCompleted`, `MissionAbandoned`, `Missions`, `CargoDepot`, and bounty events.
- Event feed/warnings: only notifications generated by monitor logic, not a raw Journal feed.
- Journal source and service status: configured folder display, selected file display, configured recent file count, Matrix status, Web status.

## Present In Journal But Not Structurally Exposed

These data are available in the local Journal or sidecar files, but the current webui cannot retrieve them as structured fields because the backend does not put them in `AppSnapshot`.

- Fuel as structured state:
  - Present: `LoadGame.FuelLevel/FuelCapacity`, `FSDJump.FuelUsed/FuelLevel`, 127 `FuelScoop` records, 428 `ReservoirReplenished` records, and `Status.json`.
  - Current webui: `HealthPanel` derives fuel text by searching `event_feed` for the latest `fuel_report` notification. If no notification is in the bounded event buffer, it shows "Fuel telemetry unavailable".
  - Gap: no `session.fuel_percent`, fuel amount, fuel source timestamp, scoop amount, or Status.json-derived live fuel.

- Ship/loadout detail:
  - Present: `Loadout.HullHealth`, `CargoCapacity`, `MaxJumpRange`, `Rebuy`, `Modules`, `ModulesValue`, `HullValue`, `ShipName`, `ShipIdent`.
  - Current webui: only ship type/localised ship name and hull damage events are surfaced.
  - Gap: no cargo capacity, jump range, rebuy, named ship identity, module list, or current loadout value.

- Credits and commander/account state:
  - Present: `LoadGame.Credits`, `Loan`, plus rank/progress fields for multiple professions.
  - Current webui: commander name and combat rank/progress may appear only inside a generated `commander_load` notification; no structured rank or credits panel.
  - Gap: no credits, loan, rank table, progress table, or game build/language.

- System/station/nav context:
  - Present: `Location` and `FSDJump` carry coordinates, allegiance, economy, government, security, population, factions, controlling power, powerplay state, conflicts, station services/economies, jump distance, and fuel used. `NavRoute.json` exists.
  - Current webui: only `session.system` is exposed.
  - Gap: no station, body, coordinates, route, distance, security, economy, faction, powerplay, conflict, jump distance, or route panel.

- Target details:
  - Present: `ShipTargeted` has `ShieldHealth` and `HullHealth` on 1,383 records, `LegalStatus` on 1,340, and `PilotRank` on 1,522.
  - Current webui: target scan events can create summary notifications, but the target health/legal/rank fields are not parsed into structured snapshot data.
  - Gap: no current target card, target shields/hull, legal status, pilot rank, or target history.

- Full mission snapshot details:
  - Present: `Missions` includes `Active`, `Failed`, and `Complete` lists in 64 snapshots. `MissionCompleted` includes `FactionEffects` on 281 records, `MaterialsReward` on 76, and donated data on 82.
  - Current backend: `MissionTracker` uses `Missions.Active` only to retain active missions. Completion/failure lists and faction/material effects are not surfaced.
  - Gap: webui cannot reconstruct mission history/effects from snapshot-only `Missions` records unless those missions were first observed through detailed `MissionAccepted` records in the running preload stream.

- Cargo/market/inventory:
  - Present: `Cargo` events and `Cargo.json`; `MarketBuy`, `MarketSell`, `Market.json`; `ShipLocker.json`, `Backpack.json`, `FCMaterials.json`.
  - Current webui: no inventory model. Trade mission delivery progress uses `CargoDepot`, but general cargo/inventory/market data are not exposed.
  - Gap: no cargo manifest, commodity prices, market inventory, backpack/locker materials, fleet carrier materials, stolen/mission cargo inventory, or trade P/L.

- Outfitting/shipyard/modules:
  - Present: `Outfitting.json`, `Shipyard.json`, `ModulesInfo.json`, `StoredModules`, `StoredShips`, module buy/retrieve/store/sell events.
  - Current webui: no module/shipyard/outfitting model.
  - Gap: no modules, stored ships/modules, shipyard availability, outfitting availability, or transfer status.

- Exploration/community/carrier/crew/social data:
  - Present in high volume: 9,006 `FSSSignalDiscovered`, 4,737 `Scanned`, 1,113 `CommunityGoal`, 995 `ShipLocker`, plus carrier, crew, wing, friend, and squadron events.
  - Current webui: not modeled except where a monitor notification happens to be emitted.
  - Gap: no exploration ledger, biology, community goals, fleet carrier finance/location/trade, crew wage/rank, wing/friend/squadron panels.

## Current Journal Missing For Existing WebUI Fields

These current webui fields or code paths exist, but the provided Journal sample does not contain source events for them:

- `FactionKillBond`-backed kills: 0 records. Combat bond kills cannot be validated from this sample.
- `MissionFailed` state: 0 records. The mission UI supports a failed state, but this sample only has completed/redirected/abandoned mission termination records.
- Structured `PowerplayMerits`: 0 `PowerplayMerits` records. `session.merits` cannot be populated from this sample; `merits_to_report` can still increase from kill bookkeeping.
- Live `Status.json` state through webui: `Status.json` exists, but current webui/backend does not read it, so flags cannot affect the displayed snapshot.

## Not A Data Source Problem

These are implementation gaps rather than missing source data:

- Sidecar files are present but unread: `Status.json`, `Cargo.json`, `Market.json`, `Outfitting.json`, `Shipyard.json`, `ShipLocker.json`, `Backpack.json`, `ModulesInfo.json`, `NavRoute.json`, and `FCMaterials.json`.
- Raw Journal events are parsed broadly, but most are kept as typed/raw events without being reduced into `SessionState`, `MissionTracker`, or `AppSnapshot`.
- `event_feed` is a bounded monitor-notification history, not an audit log. A newly opened frontend can see the backend event buffer, but only for events the backend chose to record as notifications/system events.

## Recommended Next Implementation Slice

For the next GUI/webui data expansion, prioritize fields that are present in the current Journal and useful for AFK monitoring:

1. Add structured fuel state to the backend snapshot: current fuel percent/amount, source event, updated_at. Sources should combine `LoadGame`, `FSDJump`, `FuelScoop`, `ReservoirReplenished`, and later `Status.json`.
2. Add structured current location context: system, station/body, security/economy, jump distance/fuel used, and updated_at. This uses already-present `Location`, `FSDJump`, and `Docked` data.
3. Add a target scan summary model: last scanned target, rank, legal status, target shields/hull. This requires extending `ShipTargeted` parsing to include `ShieldHealth` and `HullHealth`.
4. Add sidecar ingestion as a separate backend service layer before exposing inventory/market/status panels. Avoid putting sidecar file reads in React/Tauri frontend code.

## Verification Commands

Useful repeatable checks:

```sh
python3 - <<'PY'
import json, glob, os, collections
root=os.path.expanduser('~/Elite Dangerous')
counts=collections.Counter()
for path in glob.glob(os.path.join(root,'Journal.*.log')):
    for line in open(path, encoding='utf-8', errors='replace'):
        if line.strip():
            counts[json.loads(line).get('event')] += 1
print('lines', sum(counts.values()))
print('event_types', len(counts))
print(counts.most_common(20))
PY
```
